Safety-Critical Multi-Target Time Series Forecasting with spotforecast2 and spotforecast2-safe: A Hardened Approach to Production Machine Learning

Author
Affiliation

Thomas Bartz-Beielstein

Department of Computational Intelligence, TH Köln

Published

February 7, 2026

Abstract

Time series forecasting has become indispensable in safety-critical applications such as energy grids, financial systems, and autonomous control. However, existing machine learning libraries often prioritize generality and extensibility over determinism and auditability. We introduce spotforecast2 and spotforecast2-safe, two complementary Python packages built upon the skforecast foundation for reliable time series prediction. The spotforecast2 library provides a feature-rich environment for research and development, intelligently orchestrating hyperparameter tuning and model selection. The spotforecast2-safe package represents a deliberately minimized, hardened core suitable for safety-critical production environments, featuring zero dead code, deterministic transformations, and explicit fail-safe semantics. In this report we demonstrate multi-target forecasting on a dataset with eleven independent target columns (“A” through “K”), training a separate LGBM model for each target. Both packages implement a systematic hardening strategy that reduces the attack surface while maintaining mathematical rigor and regulatory alignment with emerging standards such as the EU AI Act. This report outlines the architectural principles, design goals, and practical implications of these systems for building trustworthy machine learning pipelines in high-risk domains.

1 spotforecast2 multi-target demo

This reference implementation demonstrates a safety-critical forecasting workflow using spotforecast2 and spotforecast2-safe on an eleven-target dataset. The primary data source is data_in.csv (located in ~/spotforecast2_data/), containing eleven independent measurement columns mapped to labels A through K. If data_in.csv is absent, the pipeline falls back to the bundled demo02.csv. A separate LGBM model is trained per target. END_TRAIN_DEFAULT is always the last timestamp in the loaded data; the model then forecasts the next 24 hours beyond that point.

The workflow executes in the following phases:

  1. Data Acquisition: Loading data_in.csv (with automatic fallback to demo02.csv) and preparing each target for the pipeline.
  2. Configuration & Model Selection: Centralized configuration of hyperparameters, feature engineering, and automated tuning strategies.
  3. Model Orchestration & Training: Fail-safe model training with automatic refitting policies, executed for each of the eleven targets.
  4. Predictive Analytics & Visualization: Generation of validated prediction packages and interactive reports for each target.

1.4 Data Acquisition

Data is loaded preferentially from data_in.csv located in ~/spotforecast2_data/ (the user data directory). If that file is not present, the pipeline falls back to the bundled demo02.csv included in the spotforecast2-safe package. Both datasets contain eleven independent target columns named A through K (column names are mapped from the instrument codes in data_in.csv). There is no forecasted load column.

The download-range constants (START_DOWNLOAD, END_DOWNLOAD) and the training cut-off (END_TRAIN_DEFAULT) are derived dynamically from the loaded dataset, so they automatically update whenever data_in.csv is refreshed.

A helper function prepare_target() writes one target column at a time to the standard interim location (interim/energy_load.csv), renaming the DateTime index to Time (UTC) and the target column to Actual Load so that all downstream pipeline components (load_timeseries(), config.data_filename) work without modification, see Table 1.

1.5 Data Processing Pipeline

The following sections implement a configurable data processing pipeline inspired by the n2n_predict_with_covariates module from spotforecast2-safe. Each step can be enabled or disabled via flags, allowing the user to choose between the original improved2 approach and the extended pipeline with exogenous covariates.

After loading the raw CSV, we aggregate/resample to hourly frequency (which also resolves any duplicate timestamps) and then validate the resulting index with basic_ts_checks(). The get_start_end() function derives the data and covariate date ranges required for exogenous feature engineering.

    agg_and_resample_data,
    get_start_end,

# The pipeline operates on a DatetimeIndex-based DataFrame with target columns only.

1.5.4 Step 3 — Missing Value Imputation with Weighting

Two imputation strategies are provided:

  • Option A — Linear interpolation (IMPUTATION_METHOD="linear"): Uses LinearlyInterpolateTS for simple linear gap filling. This is the approach used in the original improved2 workflow.
  • Option B — Weighted imputation (IMPUTATION_METHOD="weighted"): Uses get_missing_weights() which fills gaps via forward/backward fill and produces a WeightFunction that down-weights observations near former gaps during model training. This is the approach from n2n_predict_with_covariates.
from spotforecast2_safe.preprocessing import LinearlyInterpolateTS, WeightFunction
from spotforecast2_safe.preprocessing.imputation import get_missing_weights

weight_func = None  # default: no sample weighting

if IMPUTATION_METHOD == "weighted":
    logger.info("Applying weighted imputation (n2n style)...")
    df_pipeline, weights_series = get_missing_weights(
        df_pipeline, window_size=WINDOW_SIZE, verbose=True,
    )
    weight_func = WeightFunction(weights_series)
    logger.info(f"Weight function created with {len(weights_series)} entries.")
elif IMPUTATION_METHOD == "linear":
    logger.info("Applying linear interpolation (improved2 style)...")
    interpolator = LinearlyInterpolateTS()
    # LinearlyInterpolateTS expects a Series; apply per column
    for col in target_columns:
        series = df_pipeline[col]
        df_pipeline[col] = interpolator.fit_transform(series)
else:
    raise ValueError(f"Unknown IMPUTATION_METHOD: {IMPUTATION_METHOD!r}")

1.5.5 Step 4 — Exogenous Features Engineering

When USE_EXOGENOUS_FEATURES=True, the pipeline creates four categories of exogenous features using helper functions from n2n_predict_with_covariates:

  1. Weather features — temperature, wind, etc. with rolling-window aggregations.
  2. Calendar features — month, week, day-of-week, hour.
  3. Day/night features — sunrise/sunset hours, daylight duration, is-daylight flag.
  4. Holiday features — binary holiday indicators for the configured country/state.
exogenous_features = None
weather_aligned = None

if USE_EXOGENOUS_FEATURES:
    from spotforecast2_safe.processing.n2n_predict_with_covariates import (
        _get_weather_features,
        _get_calendar_features,
        _get_day_night_features,
        _get_holiday_features,
    )
    from astral import LocationInfo

    logger.info("Building exogenous features...")

    # 4a. Weather
    weather_features, weather_aligned = _get_weather_features(
        data=df_pipeline,
        start=data_start,
        cov_end=cov_end,
        forecast_horizon=PREDICT_SIZE,
        latitude=LATITUDE,
        longitude=LONGITUDE,
        timezone=TIMEZONE,
        freq="h",
        verbose=True,
    )
    logger.info(f"  Weather features: {weather_features.shape}")

    # 4b. Calendar
    calendar_features = _get_calendar_features(
        start=data_start,
        cov_end=cov_end,
        freq="h",
        timezone=TIMEZONE,
    )
    logger.info(f"  Calendar features: {calendar_features.shape}")

    # 4c. Day/night
    location = LocationInfo(
        latitude=LATITUDE,
        longitude=LONGITUDE,
        timezone=TIMEZONE,
    )

        start=data_start,
        cov_end=cov_end,
        location=location,
        freq="h",

Based on the settings shown the last code snippet and in Table 2 and Table 3, the ConfigMulti object is instantiated as follows:

  • periods: [Period(name=‘daily’, n_periods=12, column=‘hour’, input_range=(1, 24))], so that smoothed 2-hour cyclical features are generated. We utilize Radial Basis Functions (RBF) to capture seasonality. A critical design choice is the resolution of the Daily period: 12 periods are used for the 24-hour daily cycle (instead of 24). This effectively provides a 2-hour resolution, acting as a smoothing filter that reduces noise while halving the dimensionality of the feature space. The default periods are deliberately chosen to balance temporal resolution with model complexity (e.g., daily uses n_periods=12 to smooth hourly noise while reducing dimensionality).
  • lags_consider: [1, 2, 24], so that the lags of 1 hour, 2 hours and 24 hours are considered.
  • train_size: pd.Timedelta(days=365*1), so that the training window is one year. The training data always covers the year ending at END_TRAIN_DEFAULT.
  • end_train_default: END_TRAIN_DEFAULT — derived dynamically as the last timestamp in the loaded dataset. This is the cutoff separating training from the prediction horizon. Because it always equals the most recent data point, the model is always trained on the latest available year.
  • delta_val: pd.Timedelta(days=7), the size of the validation window used during hyperparameter tuning.
  • predict_size: PREDICT_SIZE = 24. The model predicts 24 steps (hours) per call.
  • refit_size: 1. Total prediction horizon = predict_size × refit_size = 24 × 1 = 24 hours. Setting this to 1 means each prediction call covers exactly the next 24 hours.

This config object is then injected into the instances of the ForecasterRecursiveModel, guaranteeing that the trained model perfectly matches the intended experimental design. For the eleven-target dataset, the same configuration is used for all targets A through K.

1.7.3 Note on explicit end_dev and train_size passage

When end_dev is None (default), handle_training calculates the training cutoff as latest data index minus 1 day. We explicitly pass end_dev=config.end_train_default (= last timestamp in the dataset) so the model is always trained right up to the most recent available observation.

We also pass train_size=config.train_size (TRAIN_SIZE = 1 year) so that train_new_model limits the lookback window to the latest 365 days ending at end_dev. This keeps training fast and ensures the model learns from recent patterns, regardless of how far back the raw CSV extends.

Since END_TRAIN_DEFAULT equals the last data point, the 24 h prediction horizon begins immediately after the last known observation — the model produces a genuine out-of-sample forecast for the next 24 hours. No future ground truth is available for comparison, so future_actual in the prediction package will be empty and metrics for the future window will not be computed.

1.7.4 Execution & Effects of Model Orchestration & Training

The entsoe_train command (via handle_training) triggers a robust, state-aware training pipeline for each of the eleven targets with the following effects:

  1. Smart Retraining Logic:
    • Check: It first checks the artifact store for an existing model of the specified type.
    • Validate: If a model exists, it calculates its age. Retraining is triggered only if the model is > 7 days old (168 hours) or if force=True.
    • Increment: If retraining occurs, the iteration counter is incremented (e.g., iteration 0 -> iteration 1).
  2. Data & Ranges:
    • Source: For each target, prepare_target() writes the corresponding column as Actual Load to energy_load.csv (with DateTime renamed to Time (UTC)), then loads data from the file specified in config.data_filename.
    • Training Window: Uses the most recent TRAIN_SIZE (1 year) of data ending at end_dev = last timestamp in the dataset. The subsequent 24-hour window is the true future forecast — no actual values are available for comparison.
  3. Artifact Storage:
    • Location: Models are serialized and saved to ~/spotforecast2_cache (or the path in SPOTFORECAST2_CACHE).
    • Format: Files are named using the convention {model}_{target}_forecaster_{n_iteration}.joblib (e.g., lgbm_A_forecaster_0.joblib, lgbm_B_forecaster_0.joblib, …, lgbm_K_forecaster_0.joblib).
    • Persistence: These artifacts are persistent and can be loaded by prediction services without retraining.
  4. Side Effects:
  • Logging: Emits INFO/ERROR logs detailing the training progress and any issues.
  • File I/O: Writes compressed .joblib files to disk.

        out["test_actual"] = test_actual

1.8.3 Execution & Effects

The prediction pipeline (entsoe_predict calling get_model_prediction and PredictionFigure) performs the following actions for each of the eleven targets:

1.8.3.1 Model Loading

Retrieves the latest trained model (highest iteration) for the given target from ~/spotforecast2_cache via get_last_model().

1.8.3.2 Prediction Package Generation

get_model_prediction() loads the latest trained model from the cache and calls package_prediction(predict_size). When end_dev equals the last dataset timestamp (genuine-future mode), package_prediction() detects that no held-out test data is available and predicts exactly predict_size steps beyond end_dev — returning an empty future_actual Series and omitting future metrics. The returned dictionary contains the components shown in Table 6.

Note: Neither data_in.csv nor demo02.csv contains a Forecasted Load column, so the keys future_forecast and metrics_forecast are not present in the prediction package.

1.8.4 Metric Computation Window

The prediction package computes metrics over two distinct windows:

  • 24h metrics (metrics_future_one_day): Calculated on the first 24 hours only (future_pred.iloc[:24])
  • Full horizon metrics (metrics_future): Calculated on the entire prediction window (predict_size * refit_size hours)

With the current configuration (predict_size=24, refit_size=1), the full horizon is 24 hours. END_TRAIN_DEFAULT equals the last timestamp in the loaded dataset, so the prediction is a genuine future forecast — no actual values exist for the prediction window. The metrics_future and metrics_future_one_day entries in the package will therefore be unavailable or empty. In-sample training metrics (metrics_train) remain available and characterise the model fit on the most recent year of data.

1.9 Training, Tuning, and Prediction Strategies

In production environments, there are different approaches to applying the models depending on computational constraints and performance requirements. Here we document and compare three distinct approaches using the same underlying dataset and configuration. Each approach is executed for all eleven targets (A through K).

1.9.1 Approach 1: Without Training and Tuning (Lazy Fitting)

By default, executing predictions via the spotforecast2_safe pipeline against an untrained configuration instantiates the base model recursively. During the very first predict() invocation, it detects the lack of a pre-fitted estimator and performs a “lazy” fit using the absolute default hyperparameters on the fly without persisting them via joblib.

This represents the absolute fastest development cycle but lacks optimization.

    def __init__(self, iteration: int, lags: int = 12, **kwargs):
        super().__init__(iteration=iteration, lags=lags, **kwargs)

    def tune(self) -> None:
        logger.info("Tuning %s Forecaster %d with spotoptim", self.name.upper(), self.iteration)

        y = load_timeseries()
        y = LinearlyInterpolateTS().fit_transform(y)
        X = self.preprocessor.build(start_date=y.index.min(), end_date=self.end_dev)

        end_train = self.end_dev - pd.Timedelta(hours=self.predict_size * self.refit_size)
        start_train = self._get_init_train(y.index.min(), end_train)
        fixed_train_size = self.train_size is not None
        length_training = len(y.loc[start_train:end_train])

        results, optimizer = spotoptim_search_forecaster(
            forecaster=self.forecaster,
            y=y.loc[start_train : self.end_dev],
            cv=self._build_cv(
                train_size=length_training,
                fixed_train_size=fixed_train_size,
                refit=False,
            ),
            search_space=search_space_spotoptim_lgbm,
            metric=self.metrics,
            exog=X.loc[start_train : self.end_dev],
            return_best=True,
            random_state=self.random_state,
            verbose=True,
            # n_trials=self.n_trials,
            n_trials=10,
            n_initial=5,
        )

        results["name"] = self.name
        self.results_tuning = results
        self.best_params = results.iloc[0].params
        self.best_lags = results.iloc[0].lags

1.9.6 Comparison of Results

A comparison of the five approaches demonstrates the trade-offs between computational overhead and model accuracy, applied independently to all eleven targets (A through K).

  • Approach 1 (Lazy Fitting) and Approach 2 (Explicit Pre-Training) produce identical predictions for each target because they both optimize the underlying tree-based estimators using default parameters.
  • Approach 3 (Optuna Tuned and Trained) explores the hyperparameter space efficiently using Optuna’s Tree-structured Parzen Estimator (TPE), typically improving the Mean Absolute Error (MAE) and corresponding Mean Absolute Percentage Error (MAPE) for all targets.
  • Approach 4 (SpotOptim Tuned and Trained) leverages the spotoptim surrogate model. It provides an alternative to Optuna that can be particularly effective with small trial budgets.
  • Approach 5 (N2N with Exogenous Covariates) uses the full data processing pipeline from n2n_predict_with_covariates, including IsolationForest outlier detection, weighted imputation with sample weighting, and exogenous covariates (weather, calendar, day/night, holidays). This approach provides the richest feature set but requires external data sources (weather API, holiday calendars).

The visualizations for each tuning approach (Approaches 3, 4, and 5) demonstrate that the tuned and enriched models provide a more nuanced fit to the patterns in each of the eleven target series compared to Approaches 1 and 2.
